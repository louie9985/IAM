## 一、项目概述

### 1.1 项目名称

IAM 统一身份与访问管理平台（Identity and Access Management Platform）。

### 1.2 项目定位

本项目不是单一登录接口，也不是只供业务系统调用的后端鉴权服务，而是公司级的统一身份治理平台。平台统一回答并记录以下问题：

- **你是谁**：账号、员工身份、组织任职、认证和多因素认证。
- **谁可以接入**：应用、OAuth Client、回调地址、授权范围和机器身份。
- **你能做什么**：应用角色、权限点、数据范围和权限模板。
- **你在哪里登录**：在线会话、设备、登录通道和强制下线。
- **当前行为是否安全**：密码策略、登录限制、IP 名单、风险规则和敏感操作二次认证。
- **谁在什么时候做了什么**：登录、授权、策略、应用接入和敏感操作审计。

平台同时提供完整的 React 可视化操作界面、标准 OAuth 2.1/OIDC 协议、OpenAPI、NestJS SDK、React SDK、权限清单工具和 RabbitMQ/Webhook 事件能力。

### 1.3 建设目标

首版建设一个可实际运营的 IAM 管理平台 MVP，而不是只建设认证主链路。八大中心均需形成可视化入口和最小业务闭环：

| 中心 | 主要职责 | 首版可视化闭环 |
|---|---|---|
| 账号中心 | 员工账号、组织、部门、岗位、用户组和 CRM 导入 | 创建、导入、调岗、停用、解锁、查询历史 |
| 认证中心 | 登录、SSO、Token、Refresh、MFA 和退出 | 登录、TOTP、授权确认、密码重置后首登改密 |
| 会话中心 | 在线用户、登录设备、Token 家族和强制下线 | 查看设备、退出设备、按用户/应用强制下线 |
| 安全中心 | 密码、登录限制、IP 名单、风险和敏感操作校验 | 策略版本、有效值预览、风险处置、Step-up |
| 权限中心 | 角色、菜单、按钮、API、数据权限和模板 | Manifest 审核、模板发布、授权追溯、紧急停用 |
| 应用中心 | Application、Client、回调、Scope、密钥和机器身份 | 接入配置、Secret 轮换、Service Principal 管理 |
| 审计中心 | 登录、操作、授权、策略和接入日志 | 检索、详情、导出、完整性校验和归档状态 |
| 开放能力 | OIDC、OpenAPI、SDK、React 组件和事件 | 文档、契约、投递状态、参考应用和接入验证 |

### 1.4 首版规模与约束

- 单公司内部平台，员工账号不超过 1,000 个。
- 持续产生请求的在线并发低于 200，首批接入应用少于 10 个。
- 一名开发者长期维护，Agent 全程辅助，无独立测试团队。
- 首版测试服务器和生产服务器各一台，均为腾讯云服务器。
- 本地、测试和生产通过 GitHub 代码仓库及不可变镜像发布链路关联。
- 核心运行组件必须可自托管并采用允许内部商业使用的开源许可证。

### 1.5 演进目标

首版按单公司模式交付，但从主键、唯一约束、租户上下文、缓存键、事件契约和 API 边界上预留：

- 集团多法人、多组织树和跨组织治理。
- SaaS/私有化产品的租户隔离、品牌配置和版本升级。
- 多节点应用、独立数据库、异机备份及 Kubernetes 部署。

首版不开放租户管理界面，不启用跨租户操作，也不为尚未发生的多租户需求引入额外运行复杂度。

## 二、项目背景与建设必要性

随着 CRM、OA、考务、财务、移动端、桌面端和外部应用逐步增加，账号、登录、权限和审计若继续由各系统分别建设，会产生以下问题：

- 同一员工拥有多套账号，入职、调岗和离职需要重复维护。
- 各应用自行实现密码、Token 和权限逻辑，安全质量无法统一保证。
- 权限分散在不同系统，无法统一追溯谁因何获得了什么权限。
- 每次业务请求都实时依赖中心权限服务时，IAM 故障会级联影响所有应用。
- 菜单、按钮和 API 权限配置与代码容易漂移，版本上线后出现误授权或误拒绝。
- 登录、授权、策略和敏感操作缺少长期、统一且可验证的审计证据。
- 新应用需要重复开发 OIDC、权限缓存、会话失效和审计上报逻辑。

本项目通过“统一身份事实源、标准认证协议、统一权限事实源、应用本地执行、可靠事件失效、全程审计”的方式解决上述问题。

## 三、产品边界与治理模式

### 3.1 两个权限管理控制面

系统采用“统一权限事实源、两个管理控制面”：

1. **IAM 全局控制面**仅供 IAM 总管理员使用，负责账号组织、应用接入、全局安全策略、权限清单审核、全局模板、跨应用审计和紧急处置。
2. **应用权限控制面**位于 CRM、OA、财务等各自系统中。应用管理员在熟悉的业务界面里创建本应用角色、组合已发布权限点并分配用户、用户组或岗位。

应用权限界面通过应用自身 BFF 调用 IAM Management API。浏览器不得持有 Client Secret，也不得直接获得全局管理权限。IAM 同时校验实际操作人、调用 Client 和目标 Application，并记录完整审计。

### 3.2 用户与角色严格分离

用户是被授权主体，角色是某个应用内的权限集合，两者不得合并建模：

- Account/Employee 不允许出现 `role`、`role_code`、`is_admin` 等角色字段。
- IAM 总管理员也通过受保护的系统角色授予，不使用隐藏布尔开关。
- 部门、岗位、用户组都不是角色。
- 只有显式 RoleAssignment 才产生权限；权限不得直接授予个人。
- 一个用户可在不同应用中拥有不同角色，角色不得跨应用混装权限。
- Token 不携带完整角色和权限集合，只携带稳定主体与授权版本。

### 3.3 后端裁决，应用本地执行

- 前端菜单隐藏和按钮置灰只改善体验，不能替代业务 API 鉴权。
- IAM 统一管理身份、角色、权限点、数据范围和授权版本。
- 业务 API 在本地验签并使用本地授权快照执行权限判断，不对每次请求同步调用 IAM。
- 数据范围由 IAM 统一表达，由业务系统根据自己的数据模型转换为查询条件。
- 权限变更通过 Outbox 和 RabbitMQ 在数秒内通知 SDK 清理缓存、拒绝旧授权版本。
- 少量高风险操作可调用 IAM Introspection 或 Step-up；IAM 不可用时此类操作失败关闭。

此模式避免 IAM 成为所有业务查询的同步单点，符合“认证核心稳定优先”的建设原则。

### 3.4 身份来源

IAM 是最终身份主档。首版支持：

- CRM 账号和组织数据一次性或分批导入。
- IAM 总管理员手工创建和维护员工账号。
- 导入前差异预检、冲突处理、幂等重试和结果报告。

企业微信、微信、钉钉、LDAP/AD 和 HR 系统连接器只预留标准接口与字段归属模型，首版不实现实际同步和第三方登录。

## 四、总体技术架构

### 4.1 架构形态

项目采用 pnpm Workspace + Turborepo 单仓库、模块化单体代码结构，并构建为三个后端运行时：

```text
React Web
  ├─ /auth/*       登录、TOTP、授权和退出
  ├─ /account/*    员工个人安全与设备
  └─ /admin/*      IAM 总管理员控制台

Nginx
  ├─ OIDC/认证路径      -> auth-runtime
  ├─ 管理/开放 API      -> control-runtime
  └─ 静态资源           -> React Web

auth-runtime
  OIDC、登录、Token、Refresh、SSO、MFA、会话关键路径

control-runtime
  账号、组织、权限、应用、安全策略、审计查询和开放管理 API

worker-runtime
  Outbox、RabbitMQ、权限失效、审计加工、导出和归档任务

基础设施
  PostgreSQL 主库 + PostgreSQL 审计库 + Valkey + RabbitMQ
```

三个运行时使用同一套领域包、同一版本和同一后端镜像，通过 `APP_ROLE=auth|control|worker` 启动不同入口。该方案不是微服务：首版不采用独立团队、独立数据库或独立发布节奏。

### 4.2 隔离原则

- `auth-runtime` 保持极简，独立限流、健康检查、资源配额和重启。
- 管理端慢查询、审计导出和后台任务不能占用认证进程资源。
- Worker 或审计库故障不阻塞登录、Token 刷新和授权配置事务。
- RabbitMQ 只承担异步通知和对外事件，不替代必须立即返回的同步领域调用。
- 关键变更与 Outbox 在同一 PostgreSQL 事务中写入，RabbitMQ 故障时事件可恢复投递。
- 首版单台生产服务器仍是主机级单点；三运行时只解决进程与代码故障隔离，不宣称主机高可用。

### 4.3 核心技术栈基线

以下为 2026-07-10 确认的搭建候选基线。项目初始化时进行最终兼容性验证并精确锁定：

| 层 | 选型 | 候选基线 |
|---|---|---|
| 运行时 | Node.js LTS | 24.x |
| 语言 | TypeScript | 5.9.3 |
| 包管理/单仓 | pnpm + Turborepo | 11.11.x + 2.10.x |
| 后端 | NestJS + Express | 11.1.x |
| OIDC 内核 | `oidc-provider` | 9.9.x |
| ORM | Prisma | 7.8.x |
| 主存储 | PostgreSQL | 18.x |
| 缓存 | Valkey，客户端 `ioredis` | 8.x / 5.11.x |
| 消息队列 | RabbitMQ | 4.x |
| 前端 | React + Vite | 19.2.x + 8.1.x |
| UI | Ant Design + ProComponents | 5.29.x + 2.8.10 |
| 路由/请求 | React Router + TanStack Query | 7.18.x + 5.101.x |
| Schema | Zod + `nestjs-zod` | 4.4.x + 5.4.x |
| API 生成 | `openapi-typescript` + `openapi-fetch` | 7.13.x + 0.17.x |

### 4.4 版本管理

- `package.json` 使用精确版本，提交 `pnpm-lock.yaml`，禁止 `latest` 和依赖自动漂移。
- Node、pnpm、容器镜像和数据库主版本全部锁定。
- 补丁版本通过完整 CI 后升级；小版本按月集中升级；大版本必须新增 ADR、迁移验证和回滚方案。
- 高危安全漏洞不等待月度窗口，优先评估和修复。
- 生成 SBOM，维护生产依赖许可证白名单。
- SDK 独立采用语义化版本；三个 IAM 运行时始终统一版本发布。

## 五、认证与会话方案

### 5.1 统一 OAuth/OIDC

所有客户端共用一个 OAuth 2.1/OIDC 协议内核，不再使用“内部自定义 JWT、外部 OIDC”双轨协议：

| 客户端 | 流程 |
|---|---|
| 内部 Web | Authorization Code + PKCE，推荐 BFF |
| 移动端/H5/桌面端 | 系统浏览器 + Authorization Code + PKCE |
| 第三方 Web | Authorization Code + PKCE + 机密客户端认证 |
| 机器身份 | Client Credentials，不签发 Refresh Token |

禁用 Implicit Grant 和 Password Grant。企业微信、微信绑定和 Device Authorization 只预留扩展点。

IAM 自身 React 管理端采用 BFF，浏览器只持有 `Secure + HttpOnly + SameSite` Cookie，不将 Access Token 或 Refresh Token 放入 `localStorage`。

### 5.2 Token 默认值

| 对象 | 默认规则 |
|---|---|
| Authorization Code | 60 秒，一次性使用 |
| 普通 Access Token | RS256 JWT，10 分钟 |
| IAM 管理端 Access Token | RS256 JWT，5 分钟 |
| ID Token | 5 分钟，仅表达认证结果 |
| Refresh Token | 256 位不透明随机值，数据库只存哈希，每次使用立即轮换 |
| 管理端会话 | 空闲 30 分钟，绝对 8 小时 |
| 移动/桌面 Refresh | 默认空闲 30 天，绝对 90 天，可由 Client 策略收紧 |
| Client Credentials Token | 5 分钟，不签发 Refresh Token |

Access Token 仅携带 `iss`、`sub`、`aud`、`client_id`、`sid`、`jti`、`tenant_id`、`auth_time`、`amr/acr` 和 `authz_version` 等稳定声明，不携带完整角色、菜单、数据范围或敏感个人信息。

Refresh Token 按令牌家族管理。旧 Refresh Token 再次出现时视为复用攻击，立即吊销整个家族和对应会话。

### 5.3 密钥与密码

- Token 从首版起使用 RS256，通过 JWKS 发布公钥。
- 签名密钥支持“下一把、当前、退休”三阶段轮换和紧急轮换。
- 密码使用 Argon2id；参数按部署机器基准测试，单次校验目标约 200–300ms。
- 默认密码最少 12、最多 128 个字符，允许粘贴和密码管理器。
- 使用本地常见弱密码词库，不调用外部密码检查 SaaS。
- 不强制定期改密，仅在泄露、管理员重置或安全事件后强制修改。
- 首版密码找回仅支持管理员重置。临时密码 24 小时内一次性有效，首次登录必须修改；重置后全部旧会话失效。

### 5.4 MFA 与敏感操作

- 首版 MFA 使用 TOTP，恢复码仅保存 Argon2id 哈希。
- 总管理员必须绑定 TOTP；普通员工由全局、应用或 Client 策略决定是否强制。
- TOTP 密钥版本化加密，验证码防同一时间步重放。
- 修改权限、策略、Client Secret、签名密钥、密码/MFA、审计导出等高风险操作要求最近 5 分钟内完成 Step-up。

### 5.5 SSO 与即时吊销

- 支持 RP-Initiated Logout、Token Revocation 和 OIDC Back-Channel Logout。
- 强制下线时吊销 Refresh Token 家族并发布 `session.revoked` 事件。
- 应用 SDK 根据 `sid` 在数秒内拒绝旧 Access Token。
- 权限变化时发布授权版本事件；旧 `authz_version` 被拒绝，客户端静默刷新。
- 不消费事件的第三方应用若要求数秒级撤权，敏感操作必须调用 Introspection；否则撤权上限为 Access Token 时效。

## 六、账号、角色与权限模型

### 6.1 账号与组织

账号、员工档案和任职关系分开建模：

```text
Account 1:1 Employee 1:N Employment
Employment -> OrganizationUnit + Position
UserGroup N:M Account
```

首版每名员工只允许一条有效任职，通过数据库约束和服务规则落实“一名员工、一个部门、一个岗位”。保留任职历史，调岗通过关闭旧任职并新建任职完成，过去审计不被改写。

账号状态为 `PENDING -> ACTIVE -> LOCKED/SUSPENDED -> DISABLED`。已有历史的账号不物理删除；离职停用同时吊销会话、终止授权并发布事件。

所有领域实体带非空 `tenant_id`，首版只存在一个隐藏默认租户。唯一约束均包含租户范围，请求和 Repository 必须携带租户上下文。

### 6.2 应用与权限点

- Application 表示逻辑业务系统，一个 Application 可有多个 OAuth Client 和 Resource Server。
- Service Principal 表示机器身份，不进入员工 Account 模型。
- 权限代码采用稳定命名，例如 `crm.customer.read`。
- 菜单和按钮是权限元数据，真正安全边界是后端 API 权限。
- 应用通过代码中的 Permission Manifest 提交候选权限版本。
- IAM 展示新增、修改、废弃和影响范围，由总管理员审核发布。
- 已被引用的权限只能先废弃、处理影响后停用，不能直接物理删除。

### 6.3 角色与模板

- 系统角色仅由 IAM 全局控制面管理。
- 应用模板角色由总管理员发布并版本化。
- 应用管理员可在本应用界面创建自定义角色，但只能组合本应用已发布权限点。
- 模板升级必须展示差异并确认，不静默修改现有角色。
- 角色可通过 RoleAssignment 分配给用户、用户组或岗位，支持生效和失效时间。
- 不允许嵌套角色，不允许权限直接授予个人，首版不引入显式 Deny。
- 功能权限按有效角色取并集；没有 Allow 即拒绝。

### 6.4 数据权限

首版标准范围为：

- `SELF`
- `ORG_UNIT`
- `ORG_SUBTREE`
- `CUSTOM_ORG_UNITS`
- `ALL`

多个角色的数据范围按业务资源取并集；`ALL` 必须显式授予。敏感字段权限独立控制并记录访问审计。IAM 返回规范化范围，业务系统在本地完成数据过滤，IAM 不理解或查询业务数据库。

## 七、可配置策略中心

### 7.1 配置原则

涉及运行规则的内容尽量策略化，但不采用无约束 EAV 或任意 JSON。每个 Policy Key 由代码声明：

- 数据类型、单位和默认值。
- 安全最小值、最大值和枚举范围。
- 允许覆盖的层级。
- 校验 Schema。
- 是否要求 Step-up。
- 是否立即影响现有会话。

数据库只保存通过 Schema 校验的版本化 Policy Set。策略生命周期为“草稿、校验、影响预览、发布、生效、发布新版本回滚”；已发布版本不能原地修改。

### 7.2 策略继承

有效值解析顺序为：

```text
代码安全默认值
-> 全局策略
-> 应用策略
-> 登录通道策略
-> OAuth Client 策略
-> 账号例外策略
```

账号例外必须填写原因和到期时间。管理界面显示每个有效值的来源、版本和覆盖原因，并提供按用户、应用、Client 预览最终策略的模拟器。

### 7.3 可配置范围

首版可配置项包括但不限于：

- 登录失败统计窗口、失败次数、渐进延迟和临时锁定时长。
- 密码长度、历史数量和临时密码时效。
- Access/Refresh Token 与会话空闲时效。
- TOTP 强制范围和 Step-up 有效时间。
- Web、Mobile、Desktop 的并发设备数及超限动作。
- 新设备、IP 变化、风险阈值和阻断规则。
- IPv4/IPv6 CIDR 允许/拒绝名单。
- Client Secret、签名密钥轮换周期。
- 授权缓存、Webhook 重试和审计在线保留周期。

租户隔离、回调地址精确校验、PKCE、审计不可删除、密码哈希最低强度、敏感信息不写日志和后端鉴权属于安全不变量，不提供关闭开关。

### 7.4 分登录通道的设备限制

每个 OAuth Client 注册时必须指定可信登录通道：`WEB`、`MOBILE`、`DESKTOP` 或 `SERVICE`。不依靠可伪造的 User-Agent 判断通道。

并发限制按 `(账号, 应用, 登录通道)` 分别计算。默认 Web 最大 1 台、Mobile 最大 1 台、Desktop 最大 1 台，因此一台电脑和一台手机可以同时登录。

达到上限时默认展示现有设备，由用户确认后下线最早会话并继续登录。该动作可按策略改为拒绝新登录、下线全部同通道会话等。浏览器设备实际表示浏览器配置文件和 SSO 会话，不宣称能精确识别物理硬件。

## 八、前端可视化设计

### 8.1 单一 React 应用

一个 React 工程承载：

- `/auth/*`：登录、TOTP、授权确认、退出和协议错误。
- `/account/*`：所有员工可用的密码、MFA、个人会话和设备。
- `/admin/*`：仅 IAM 总管理员可用的管理控制台。

管理端按工作任务分组导航，而不是直接平铺后端模块：

| 导航组 | 页面范围 |
|---|---|
| 运行概览 | 服务状态、可用性、待审核、风险、积压和审计延迟 |
| 身份与组织 | 员工、组织树、岗位、用户组、CRM 导入 |
| 访问控制 | 权限目录、模板、应用角色查看、授权、Manifest 审核 |
| 安全运营 | 会话、设备、锁定、风险、策略、IP 名单、策略模拟器 |
| 平台接入 | 应用、Client、回调、Scope、机器身份、密钥轮换 |
| 开放能力 | OpenAPI、SDK、事件、Webhook/RabbitMQ 投递状态 |
| 审计与合规 | 检索、详情、导出、归档和完整性校验 |

### 8.2 交互规则

- 统一列表、详情、表单、差异查看和异步任务状态。
- 筛选、分页和详情状态进入 URL，可刷新、返回和深链。
- 高风险操作展示影响范围、要求填写原因并触发 Step-up。
- Secret、恢复码只显示一次，离开页面后不可再次取得明文。
- 策略页展示继承来源、最终有效值、版本差异和影响预览。
- 审计页面只读，不提供修改和删除入口。
- 错误提示提供可操作说明和 `trace_id`，不暴露内部堆栈。
- 管理端以桌面操作为主，768px 仍可用；公开认证和个人安全页面完整适配手机。

### 8.3 状态边界

- TanStack Query 管理服务端状态。
- URL 管理筛选、分页和视图状态。
- React Context 仅保存当前身份、主题和环境信息。
- 首版不引入 Redux 或 Zustand。
- 前端禁止硬编码角色判断，统一使用权限 Hook。

## 九、审计与长期保留

### 9.1 事务内留痕，事务外加工

关键业务变更与 Audit Intent、Outbox Event 在同一主库事务中写入。Worker 异步将审计事件写入独立审计 PostgreSQL：

- 审计库或 RabbitMQ 故障不阻塞登录和管理事务。
- Worker 按 `event_id` 幂等补写。
- 敏感数据读取必须先成功生成审计意图，否则失败关闭。
- 密码、Token、Client Secret、TOTP 密钥和恢复码不得进入审计。

### 9.2 在线检索与归档

- 审计库使用独立实例、Volume 和凭据，按月分区。
- Writer 只允许插入，Reader 只允许查询。
- 默认最近 24 个月在线可检索。
- 每日生成压缩、加密的 JSONL 归档包、Manifest 和签名。
- 使用 Merkle Root 和独立审计签名密钥检测修改、删除或重排。
- 导出提供 CSV 和 JSONL，加密并限时下载；导出行为本身也被审计。

首期归档文件仍位于生产服务器，只能检测篡改，不能抵御主机 root 权限或整机损坏。COS 归档驱动、对象版本和 Object Lock 作为后续能力预留，首期不启用，也不宣称已经具备异机不可变归档。

## 十、开放能力与接入规范

### 10.1 SDK

| 包 | 职责 |
|---|---|
| `@iam/contracts` | API、事件、策略和权限公共契约 |
| `@iam/core` | 框架无关 OIDC、JWKS、Token 校验和 API Client |
| `@iam/nestjs` | Guard、Decorator、授权缓存、事件消费和审计 Outbox |
| `@iam/react` | AuthProvider、Hooks、PermissionGate 和路由守卫 |
| `@iam/manifest` | 权限 Manifest 定义、校验、差异提交 CLI |

SDK 只能使用公开协议和 API，不允许连接 IAM 数据库或导入内部领域模块。

### 10.2 事件

事件采用 CloudEvents 1.0 和版本化 JSON Schema。内部应用使用 RabbitMQ 持久队列，外部应用使用签名 Webhook。投递语义为至少一次，消费者按 `event_id` 幂等，并根据聚合版本检测乱序和丢失。

首版核心事件：

```text
iam.identity.account.changed.v1
iam.identity.employment.changed.v1
iam.authorization.subject.changed.v1
iam.authorization.manifest.published.v1
iam.session.revoked.v1
iam.security.policy.published.v1
iam.application.client.changed.v1
```

### 10.3 参考应用

仓库包含一个最小 NestJS + React 参考应用，持续验证登录、BFF、权限 Guard、数据范围、应用角色管理、即时收权、会话吊销和审计上报，作为接入方的可执行文档。

## 十一、工程结构与 Agent 治理

### 11.1 前后端物理分区

```text
apps/frontend/web              React 应用
apps/backend/*-runtime         三个薄运行入口
packages/backend/modules/*     领域模块
packages/backend/platform/*    无业务规则的平台能力
packages/contracts             公共契约及生成产物
packages/sdk/*                 对外 SDK
```

前端只能依赖 Contracts 和前端 SDK，禁止导入后端包。后端 Runtime 只负责装配，禁止直接编写领域逻辑。

### 11.2 领域所有权

每个领域模块独立 Workspace 包，包含：

- `domain/application/infrastructure/presentation` 分层。
- 唯一公共 `index.ts`。
- `module.yaml`：风险等级、表所有权、允许依赖和事件。
- `README.md`：职责、非职责、接口、不变量和测试命令。
- 红类模块的局部 `AGENTS.md`。

路径表达所有权，公共入口表达依赖，`module.yaml + CI` 表达禁止事项。禁止跨模块直接访问 Repository 或数据表，禁止创建业务逻辑大杂烩 `common/utils/helpers`。

### 11.3 Agent 风险分级

- **红类**：OIDC/Token、密码/MFA、租户隔离、授权解析、策略引擎、会话吊销、密码学、审计完整性和数据库迁移。测试先行，人工逐行审核。
- **黄类**：管理 API、CRM 导入、事件消费者、SDK 和策略 UI。要求集成测试和契约评审。
- **绿类**：展示组件、只读页面、文档和生成代码。自动门禁通过后常规走查。

Agent 任务必须包含事实源、范围内/范围外、验收条件、可修改目录、契约影响、威胁点、测试命令和回滚方式。一个任务只完成一个可验证的领域行为。

### 11.4 文档事实源

- 本立项书定义目标、范围、资源、里程碑和验收。
- `docs/specs` 保存已审批总体设计规格。
- `docs/product` 保存各领域产品规则。
- `docs/architecture` 保存架构与 ADR。
- `packages/contracts/src` 是 API Schema、事件和策略定义的人工事实源。
- OpenAPI、事件 JSON Schema 和 SDK Client 自动生成，禁止手工修改。
- 根目录只保留 `AGENTS.md`、`README.md` 和工程配置，减少噪声。

## 十二、测试、安全与质量门禁

### 12.1 测试分层

- Vitest + React Testing Library：领域单元和 React 组件测试。
- Testcontainers：真实 PostgreSQL、Valkey 和 RabbitMQ 集成测试。
- Playwright：登录、MFA、设备限制、权限审核、即时收权、下线和审计主链路。
- OIDC Conformance：标准协议一致性。
- OWASP ZAP、Gitleaks、Trivy 和 OSV Scanner：Web、密钥、镜像和依赖扫描。
- 故障注入：Valkey、RabbitMQ、Worker 和审计库故障下的预期降级。

红类核心逻辑分支覆盖率不低于 90%，其他后端核心不低于 80%。覆盖率不能替代威胁模型和关键失败用例。

### 12.2 CI 门禁

```text
PR：
format -> lint -> boundary -> typecheck
-> unit -> contract drift -> secret/dependency scan

main：
Testcontainers -> Compose E2E -> Playwright
-> image scan -> 部署测试服务器

release：
OIDC Conformance -> ZAP -> 故障测试
-> 迁移预检 -> 恢复验证 -> 人工批准生产
```

### 12.3 数据库迁移

Prisma Migration 可由 Agent 辅助生成，但 SQL、锁表影响、回填、回滚和备份条件必须人工审查。生产迁移需人工批准，采用 Expand–Migrate–Contract，避免新旧版本不兼容。

## 十三、环境、发布与运维

### 13.1 三套环境

| 环境 | 用途 | 数据与部署 |
|---|---|---|
| 本地 | 开发、调试、自动测试 | Docker Volume 本地暂存，不承诺 RPO/RTO |
| 测试服务器 | 集成、E2E、安全验证、演示 | 腾讯云单服务器，与生产数据和密钥完全隔离 |
| 生产服务器 | 正式运行 | 腾讯云单服务器，Docker Compose、健康检查和自动重启 |

三套环境使用不同数据库、密钥、域名、OAuth Client、回调地址和管理员账号。真实密钥只存在对应环境，不进入 GitHub；仓库仅提交环境变量模板。

### 13.2 GitHub 与不可变镜像

```text
本地分支提交
-> GitHub PR/CI
-> 合并 main
-> 按 Commit SHA 构建一次 GHCR 不可变镜像
-> 测试服务器拉取并验收
-> 创建 Release Tag
-> 生产服务器按同一镜像摘要部署
```

测试和生产必须使用同一个镜像摘要。服务器可保留 Git 工作区用于 Compose、迁移和部署脚本，但只检出明确 Commit/Release Tag，禁止 `git pull main` 后现场构建生产应用。回滚使用上一 Release Tag 和镜像摘要。

### 13.3 可用性与恢复目标

- 运营目标：月度可用性 99.9%，计划维护窗口每月不超过 2 小时。
- IAM 故障时，已登录业务应用依靠本地验签和授权缓存继续普通操作。
- 新登录、Refresh、权限变更和高风险操作失败关闭。
- 目标 RTO 不超过 2 小时；目标 RPO 不超过 5 分钟。

但首版生产仅单服务器且 COS 备份未启用，因此上述 99.9%、RTO 和 RPO 是运营与增强目标，不是可严格保证的 SLA。整机损坏时可能无法满足目标。

首版预留备份存储驱动和腾讯云 COS 配置契约，但不实际接入。后续启用 PostgreSQL 全量备份、WAL 持续归档、COS 版本/Object Lock 和恢复演练后，再将 RPO/RTO 转为正式验收指标。

## 十四、里程碑与资源计划

### 14.1 研发资源

- 一名全栈开发者负责架构、前后端、测试、部署和维护。
- Agent 用于规格辅助、代码生成、测试补全、审查和文档同步。
- 红类代码、数据库迁移和生产发布必须人工批准。
- 无独立测试与运维人员，因此自动化门禁、测试服务器验证和 Runbook 属于首版交付物，而非可选项。

### 14.2 分批交付

| 阶段 | 范围 | 出口标准 |
|---|---|---|
| M0 工程底座 | Monorepo、三运行时、前端壳、基础设施、CI、契约生成 | 本地与测试环境可重复启动，边界检查和镜像链路跑通 |
| M1 身份认证 | 账号组织、CRM 导入、OIDC、密码、TOTP、会话和策略 | 员工可登录、刷新、退出，管理员可导入、重置、下线和配置通道限制 |
| M2 应用权限 | Application、Client、Manifest、模板、角色、数据范围、机器身份 | 参考应用完成权限管理、本地鉴权和数秒级收权 |
| M3 平台闭环 | 八大中心界面、独立审计库、SDK、事件、导出和归档 | 总管理员可完成全部治理闭环，审计故障不影响认证 |
| M4 上线验证 | E2E、安全扫描、OIDC 一致性、故障演练、迁移演练、CRM 试接入 | 所有验收链路通过，测试镜像按同一摘要晋级生产 |

按一名全职开发者加 Agent 的资源模型，建议规划基线为 24–32 周。该区间包含测试服务器联调和 CRM 试接入，不包含新增短信、企业微信实际登录、COS 备份或多节点高可用。

## 十五、首版验收标准

1. CRM 导入员工与组织，预检差异、冲突处理和幂等重试均可验证。
2. 员工完成 Code + PKCE 登录、TOTP、Refresh 轮换、退出和单点退出。
3. 管理员重置密码后旧会话失效，临时密码首次登录必须修改。
4. Web 与 Mobile 并发限制分别生效；一台电脑和一台手机可同时登录，第二台同通道设备按策略处置。
5. 应用提交 Manifest，IAM 展示差异并由总管理员审核发布。
6. 应用管理员在 CRM 自身界面创建角色并分配员工，不能跨 Application 操作。
7. Account/Employee 不含角色字段，权限只能通过 RoleAssignment 产生。
8. 业务 API 本地验签和执行数据范围，普通请求不同步依赖 IAM。
9. 权限撤销与强制下线通过事件在内部参考应用数秒内生效。
10. 第三方 Web 完成标准 OIDC 授权码流程，机器身份完成 Client Credentials。
11. 安全策略可按全局、应用、通道、Client 和账号例外解析、预览、发布和回滚。
12. 登录失败、风险规则、IP 名单和敏感操作 Step-up 按有效策略执行。
13. 登录、授权、策略、应用和敏感操作审计可检索、导出并验证完整性。
14. RabbitMQ、Worker 或审计库故障时认证主链路仍可运行，事件可在恢复后补偿。
15. GitHub 构建一次不可变镜像，测试验收后的同一镜像摘要成功部署生产。
16. CI 的类型、边界、契约、测试和安全扫描全部通过，红类变更有人工作出审核记录。

## 十六、主要风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 生产服务器为单点 | 整机故障导致 IAM 中断 | 三运行时隔离进程风险；后续启用 COS/WAL 和多节点，不虚假承诺高可用 |
| 单人团队范围过大 | 八中心 MVP 延期或质量下降 | 按 M0–M4 交付，严格控制每中心深度，高级风控和第三方连接器延后 |
| OAuth/OIDC 实现错误 | 身份伪造或会话泄露 | 使用成熟 `oidc-provider`，运行 Conformance，红类人工审核 |
| 策略过度配置 | 配置冲突、误锁定或安全降级 | 有类型策略目录、安全上下限、影响预览、版本化和安全不变量 |
| 权限事件丢失或乱序 | 应用继续使用旧权限 | Outbox、至少一次投递、event_id 幂等、聚合版本和增量恢复 |
| 应用管理员越权 | 跨应用修改角色或授权 | 同时校验操作人、调用 Client、目标 Application；所有写操作审计 |
| Agent 跨边界生成代码 | 领域耦合和安全回归 | 物理目录、公共入口、module.yaml、CI boundary、风险分级和小任务 |
| 审计长期增长 | 主库/查询变慢 | 独立审计库、按月分区、异步导出和长期归档 |
| 生成契约漂移 | 前后端和 SDK 不一致 | 单一 Schema 事实源、自动生成、CI drift 检查和参考应用 |
| 首期无异机备份 | 整机损坏可能丢失数据 | 明确风险；预留 COS 驱动，启用前不承诺正式 RPO/RTO |

## 十七、明确不在首版范围

- 短信、邮箱验证码和自助找回密码。
- 企业微信、微信、钉钉、LDAP/AD、HR 的实际同步或登录。
- 外部人员、访客、临时人员账号。
- 多租户管理界面、跨租户操作和 PostgreSQL RLS。
- 八中心独立微服务、Kubernetes 和多节点高可用。
- 机器学习风险模型、设备指纹和逐记录 ABAC 策略语言。
- ClickHouse、OpenSearch/Elasticsearch 等专用审计分析库。
- 腾讯云 COS 实际备份与不可变归档。
- 短信、Sentry 等付费或托管 SaaS 依赖。

## 十八、立项结论

项目采用“统一 OIDC 协议内核、统一身份与权限事实源、应用本地执行、数秒事件收权、三运行时隔离、完整可视化、长期审计、Agent 可检查工程边界”的总体方案。

该方案在当前 1,000 用户、少量应用、一名开发者和单服务器部署条件下，优先保障协议正确、安全可控、认证稳定和长期可维护；同时为集团化、产品化、多节点和异机归档保留明确演进路径。

本立项书经最终审阅后，作为后续详细计划、工程骨架、各领域产品规格、API/事件契约和实施验收的上游依据。任何改变统一认证协议、用户角色分离、权限本地执行、策略安全不变量、三运行时隔离或审计事实生成方式的变更，必须新增 ADR 并重新评估风险。
