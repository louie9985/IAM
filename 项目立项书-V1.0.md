# IAM 统一认证权限中心 项目立项书 V1.0

> 用途：供老板/决策会审批立项。本立项书从**定位层、框架层、模块层、功能层、决策层、指标层**说明这套系统"替谁裁决、谁来接入、谁担责、谁受益"，即统一身份治理视角下的系统架构与建设路径。
> 事实源：`IAM 功能架构图 V1`、`认证鉴权模块设计文档 docs/01~04`、`ZSJ-CRM 项目立项书 V3.1（身份与组织平台组）`。本文如与后续受控技术文档冲突，以受控文档为准。
> 关联项目：本系统由 ZSJ-CRM 中"身份与组织平台组"**抽离上升**为公司级独立基础设施。CRM 现有账号体系迁入 IAM 统一治理，CRM 自身不再维护独立登录鉴权，改为接入 IAM。后续新增业务系统同此模式。

---

## 一、项目背景与立项目的

随着公司业务系统从单一 CRM 扩展到多套(CRM、OA、考务、财务、外部提交端等)，"身份"正在各系统中被重复建设：每个系统各自维护一套账号密码、各自配一套权限、各自记一套日志。这直接带来四个无法回避的问题：

- **账号分散**：同一个员工在不同系统有不同账号，入职要开 N 次号，离职要关 N 次号，漏关一个就是安全隐患。
- **权限孤岛**：谁在哪个系统有什么权限，没有一个地方能统一看清、统一收回，权限审计无从谈起。
- **重复登录**：员工在系统间跳转要反复登录，第三方应用接入没有标准协议，每接一个都要定制。
- **责任无痕**：登录、授权、权限变更这些最敏感的动作散落各处，出了安全事件无法统一追溯。

本项目要解决的不是"再做一个登录页"，而是**把"你是谁、你能做什么、你干过什么"这三件事从各业务系统里抽出来，收归一处统一裁决、统一留痕**。业务系统从此不再自建账号权限体系，只负责"拿着 IAM 发的身份去判断业务"，身份治理的责权利边界一次性对齐到公司级基础设施。

**立项目的**：建设 IAM 统一认证权限中心，作为公司所有业务系统的**统一身份底座**——对内实现"一处登录、全线通行、统一授权、全程留痕"，对外提供标准协议与 SDK 支撑第三方接入，为公司从"多个孤立系统"走向"统一数字化平台"打基础。

## 二、系统定位与责权利体系（顶层设计）

IAM 与业务系统的关系，不是"平级的另一个系统"，而是**所有系统之上的裁决层**。它自己不承载任何业务，只回答三个问题，并为这三个答案担全责。

### 2.1 三问定位：认证 / 授权 / 审计

| 定位 | 回答的问题 | 对应中心 | 责权利定位 |
|---|---|---|---|
| 认证(你是谁) | 证明请求者确为本人 | 认证中心 + 会话中心 | 对"身份真实性"负全责，是所有系统信任的起点 |
| 授权(你能做什么) | 判定某操作是否被允许 | 权限中心 | 对"权限判定一致性"负责，业务系统不得自行绕过 |
| 审计(你干过什么) | 记录谁在何时做了什么 | 审计中心 | 对"关键动作可追溯"负责，只记录不干预业务 |
| 护栏(全程防御) | 阻断异常与风险行为 | 安全中心 | 横切在认证/操作各环节，对"账号与资金安全"负责 |
| 接入(谁能进来) | 管理哪些应用可接入 | 应用中心 | 对"接入方身份与回调安全"负责，防止钓鱼与越权 |
| 输出(能力外供) | 把上述能力交给业务系统用 | 开放能力 | 对"接入标准统一"负责，业务系统按标准对接 |

### 2.2 一条铁律:后端裁决，前端不代权

身份与权限的判断**一律由 IAM 后端裁决**，业务系统的前端只做展示与提示。菜单隐藏、按钮置灰是"体验"，不是"安全"——真正的边界永远落在后端的 API 与数据权限校验上。这条铁律与 ZSJ-CRM 立项书中"前端界面不能代替权限判断"完全一致，本项目将其上升为公司级统一约束。

### 2.3 责任归属基于"当时"，不是"现在"

与 CRM 立项书中"业绩归属按成交时点判定"同理:IAM 的每一条审计记录锚定**动作发生时**的身份与权限快照。用户转岗、离职、权限调整，都不改写历史记录——**"谁在当时有权做了什么"永远可追溯**，从根本上杜绝离职甩锅、事后抵赖。

### 2.4 主体来源:账号中心是唯一户籍库

系统内一切身份的源头是账号中心(用户/组织/部门/岗位/用户组)。权限不直接配给个人，而是通过**角色、岗位、用户组**批量授予——人进组自动继承，人离组自动收回。这保证了"入职开一次号、离职关一次号"，从建模上消灭账号分散问题。

## 三、系统架构（框架层）

系统分**平台底座**与**认证业务中心**两层，边界铁律:业务中心之间不直接互相调用，只通过统一事件机制(RabbitMQ)交互；审计等消费型中心只订阅事件、不反向阻塞主链路。这一约束对应组织上"认证核心必须极致稳定、其余能力解耦演进"的要求。

### 3.1 技术底座

| 底座 | 选型 | 承担的职责 |
|---|---|---|
| 应用框架 | NestJS (TypeScript) | 统一的模块化服务框架 |
| 主存储 | PostgreSQL + Prisma | 用户、应用、角色权限等强一致实体 |
| 会话/缓存 | Redis | 令牌、在线会话、登录限流、黑名单 |
| 消息总线 | RabbitMQ | 认证域事件外发，供审计/风控/业务系统订阅 |
| 协议内核 | node-oidc-provider(OIDC) | 标准 OAuth2/OIDC 协议，经 OpenID 官方认证 |

### 3.2 双轨令牌策略

IAM 对不同接入方采用两条并行的身份通道，避免"一套方案硬套所有场景":

- **内部 first-party 应用**:走轻量令牌通道——`JWT access(短时效，无状态校验保性能) + 不透明 refresh(Redis 存储，可即时吊销)`。高频业务请求走无状态快路径，强制下线/单点退出靠 refresh 吊销 + access 黑名单兜底。
- **第三方/需标准协议的应用**:走标准 OIDC 通道——由 `node-oidc-provider` 提供协议内核，IAM 只实现"接谁的账号、用谁的存储、塞谁的权限"三个适配点，协议正确性由官方认证的库保证，不自研踩安全坑。

两条通道共用同一套账号、会话、权限、审计，对业务侧是统一的身份来源。

## 四、模块层（八大中心与耦合度分层）

八大中心按**耦合度与稳定性要求**分三层建设，而非一次性堆成单体。这是本项目最关键的架构判断:

```
核心层(极致稳定、高性能、少变更 —— 独立服务先行)
  认证中心 + 会话中心 + 安全中心(登录基础)
业务层(规则多、变更频繁 —— 独立演进,事件/API 解耦)
  权限中心 + 应用中心
消费层(纯订阅、挂了不影响主链路 —— 完全独立)
  审计中心
输出层(对外封装 —— 随核心稳定后逐步开放)
  开放能力
```

| 中心 | 职责边界 | 耦合层 |
|---|---|---|
| **账号中心** | 用户/组织/部门/岗位/用户组——身份的唯一户籍库 | 核心依赖 |
| **认证中心** | 登录/SSO/Token/Refresh/单点退出/MFA——证明"是本人" | 核心层 |
| **会话中心** | 在线用户/登录设备/Token 管理/强制下线——管在线状态 | 核心层 |
| **安全中心** | 密码策略/登录限制/IP 名单/风险登录/敏感操作校验——全程护栏 | 核心层(基础)+ 独立(风控) |
| **权限中心** | 角色/菜单/按钮/API 权限/数据权限/权限模板——判定"能做什么" | 业务层 |
| **应用中心** | 应用管理/Client/回调/授权范围/接入密钥——管"谁能接入" | 业务层 |
| **审计中心** | 登录/操作/权限变更/应用接入日志——记录"干过什么" | 消费层 |
| **开放能力** | OpenAPI/NestJS SDK/React 权限组件/RabbitMQ 事件——能力外供 | 输出层 |

**为什么这样分层**:认证与会话是所有系统信任的地基，必须先行且保持稳定；权限规则会随业务频繁调整，独立出去才不会拖累认证核心；审计是纯消费方，即使故障也绝不能阻塞用户登录；开放能力等核心稳定后再逐步对外，避免早期接口反复变更伤及接入方。

## 五、功能层（责权利在具体操作上的落地原则）

功能层不逐页罗列，只锁定统一原则:

1. **谁能看**:数据权限(本人/本组/本部门/全公司)由 IAM 后端统一判定并下发，业务系统前端不得以隐藏代替鉴权。
2. **谁能做**:API 权限是唯一的安全边界。任何写操作，业务系统都须持 IAM 令牌向权限中心校验，前端置灰不构成防护。
3. **谁担责**:登录、授权、强制下线、权限变更等关键动作全部记录操作人、时间、身份快照，异步落审计中心，支持事后追溯与申诉。
4. **谁受益**:外部提交者/兼职等非员工角色单独设计分层可见规则,通过应用中心的授权范围(scope)精确限定其可访问的能力,做到"该给的给、不该碰的碰不到"。

## 六、关键技术决策（已定与待定）

本项目在技术选型上有若干关键判断，已在设计阶段收敛，立项层面明确记录:

| 决策点 | 结论 | 理由 |
|---|---|---|
| OIDC 协议实现 | **成熟内核(node-oidc-provider)+ NestJS 外壳**，不自研 | 协议是死标准，自研收益为零、安全风险全归自己；库经 OpenID 官方认证 |
| 令牌形态 | JWT access + 不透明 refresh 双令牌 | 高频请求无状态保性能，refresh 可吊销保可控(强制下线/单点退出) |
| 强制下线机制 | 吊销 refresh + access 黑名单(TTL 自动清理) | 兼顾即时性与存储可控 |
| 模块解耦 | 中心间只通过 RabbitMQ 事件交互 | 审计/风控挂掉不影响登录主链路 |
| 首期范围 | 认证核心主链路先行，权限/应用/审计分批 | 控制首版复杂度，让地基先稳 |
| 数据库/ORM | PostgreSQL + Prisma | 类型安全、迁移体验好 |

## 七、里程碑与分批交付

采用**核心先行、分批解耦**的建设节奏，而非一次性硬切。各批次出口以"可端到端演示"为准:

| 批次 | 内容 | 出口标准 |
|---|---|---|
| B1 认证核心 | 账号中心(基础)+ 认证中心(密码登录/JWT/Refresh)+ 会话中心(在线/强制下线)+ 安全中心(登录限流/失败锁定) | 能建用户、登录发令牌、刷新、退出、单点退出、管理员强制下线全通 |
| B2 权限中心 | 角色/菜单/按钮/API 权限/数据权限/权限模板 + 权限校验中间件 | 配角色→绑权限→令牌携带角色→API 鉴权→数据范围过滤全通 |
| B3 应用中心 + SSO/OIDC | 应用管理/Client/回调/scope/接入密钥 + node-oidc-provider 接入 | 第三方应用走标准 authorization_code + PKCE 登录并拿到 id_token |
| B4 安全增强 + 审计中心 | MFA/密码策略/IP 名单/风险登录 + 审计中心事件消费落库 | 高风险登录触发 MFA；登录/权限变更日志可查 |
| B5 开放能力 | OpenAPI + NestJS SDK + React 权限组件 + 事件契约 | 一个业务系统(如 CRM)完整接入 IAM，撤掉自建账号体系 |

> 具体每批周期与起止日期待第八节资源确定后倒排;交付顺序遵循"地基稳了再往上盖"。

## 八、组织与资源投入

- **研发模式**:延续 ZSJ-CRM 模式——小规模全栈团队 + AI 全量辅助编码，无独立测试团队，测试责任下沉到核心模块单元测试(令牌轮换、权限判定等关键逻辑)+ 关键链路人工验证。
- **验收人分工**:认证/会话由技术负责人验收;权限模型由各业务系统负责人共同验收(因权限最终服务于业务);安全策略由安全/运维负责人验收。

## 九、风险登记

| 风险 | 影响 | 对策 |
|---|---|---|
| 认证核心是全公司单点，故障即全线登录不可用 | 一旦挂掉所有系统无法登录 | 核心层保持极简稳定;Redis 故障时 access 仍可无状态校验(牺牲即时吊销换可用性)并告警;关键路径高可用部署 |
| 令牌/协议实现的安全漏洞(隐性、可登录但可被攻击) | 身份被伪造，后果最严重 | OIDC 用官方认证库不自研;令牌方案遵循业界主流折中;安全逻辑重点单测 |
| 存量业务系统迁移到 IAM 的改造成本 | CRM 等已自建账号，切换需改造 | 提供 SDK 降低接入成本;双轨令牌允许渐进迁移，老系统不强制一次切净 |
| 权限模型随业务频繁变更 | 权限规则滞后于实际管理需求 | 权限中心独立演进、事件解耦，变更不影响认证核心 |
| 单点登出/会话一致性在多系统间的复杂度 | 一处退出，他系统未同步下线 | 首期先保内部 SSO;back-channel logout 随 OIDC 批次落地 |

## 十、验收标准

以下端到端主链路作为整体验收基准，每条是一次完整的身份治理闭环:

1. 建用户→密码登录→签发双令牌→携带 access 访问受保护接口→access 过期→refresh 换新→退出(验证认证与会话闭环)
2. 配角色→绑 API/数据权限→登录令牌携带角色→越权 API 被拦截→数据范围正确过滤(验证授权闭环)
3. 管理员强制下线某设备→该设备 access 与 refresh 同时失效→需重新登录(验证会话控制闭环)
4. 注册第三方应用→standard OIDC 授权码+PKCE 登录→回调白名单校验→拿到 id_token(验证接入闭环)
5. 登录/权限变更等动作→异步落审计中心→按操作人/时间可查→审计中心故障不影响登录(验证留痕与解耦闭环)

各批次以第八节验收人签字确认为准;全部通过后 IAM 具备承接首个业务系统(CRM)接入的条件。
